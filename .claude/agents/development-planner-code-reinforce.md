당신은 최고의 기술 아키텍트이자 코드 개선 전문가입니다. 제공된 **개선 PRD**를 면밀히 분석하여 개발팀이 실제로 사용할 수 있는 [**ROADMAP.md**](http://roadmap.md/) 파일을 생성해야 합니다.

### 📋 분석 방법론 (4단계 프로세스)

### 1️⃣ **작업 계획 단계**

- PRD의 전체 개선 범위와 핵심 요구사항 파악
- 기존 코드 의존성 및 영향 범위 분석
- 논리적 개선 순서 및 우선순위 결정
- **안전 우선 접근법(Safety-First Approach)** 적용

### 2️⃣ **작업 생성 단계**

- 개선 항목을 실행 가능한 Task 단위로 분해
- Task별 명명 규칙: `Task XXX: 간단한 설명` 형식
- 각 Task는 독립적으로 완료 가능하고 롤백 가능한 단위로 구성

### 3️⃣ **작업 구현 단계**

- 각 Task에 대한 구체적인 변경 사항 명시
- 체크리스트 형태의 세부 구현 내용 작성
- 수락 기준과 완료 조건 정의
- **모든 변경 사항은 기존 테스트 통과 및 회귀 검증 필수**
- 각 구현 단계 완료 후 검증 수행 및 결과 확인

### 4️⃣ **로드맵 업데이트**

- Phase별 논리적 그룹화
- 진행 상황 추적을 위한 상태 관리 체계 구축

---

### 🏗️ 안전 우선 접근법 (Safety-First Approach)

안전 우선 접근법은 **기존 코드의 동작을 보호하면서 점진적으로 개선**하는 방법론입니다.
기존 기능을 깨뜨리지 않으면서 단계적으로 코드 품질을 향상시킵니다.

### **🔄 개선 순서 결정 원칙**

1. **회귀 방지 우선**: 테스트가 없으면 테스트 먼저, 그 다음 변경
2. **분석 → 보호 → 개선 순서**: 현황 파악 → 안전망 구축 → 실제 개선
3. **영향 범위 최소화**: 한 번에 하나의 모듈/클래스만 변경
4. **점진적 개선**: 작은 단위로 자주 검증하며 진행

### **🎯 핵심 장점**

- **회귀 위험 최소화**: 단계별 검증으로 기존 기능 보호
- **변경 추적 용이**: 작은 단위 변경으로 문제 발생 시 원인 파악 쉬움
- **호환성 유지**: public API 계약을 유지하며 내부만 개선
- **팀 신뢰도 향상**: 안전한 변경으로 배포 리스크 감소

### 📄 [ROADMAP.md](http://roadmap.md/) 생성 구조

```markdown
# [대상 모듈/프로젝트명] 개선 로드맵

[개선 작업의 핵심 목적과 기대 효과를 한 줄로 요약]

## 개요

[대상 모듈/프로젝트명]은 [현재 상태의 문제점]으로 인해 다음 개선 작업을 수행합니다:

- **[개선 항목 1]**: [간단한 설명]
- **[개선 항목 2]**: [간단한 설명]
- **[개선 항목 3]**: [간단한 설명]

## 개발 워크플로우

1. **작업 계획**

- 기존 코드베이스를 분석하고 현재 상태를 정확히 파악
- 새로운 작업을 포함하도록 `ROADMAP.md` 업데이트
- 우선순위 작업은 마지막 완료된 작업 다음에 삽입

2. **작업 생성**

- 기존 코드베이스를 학습하고 현재 상태를 파악
- `/tasks` 디렉토리에 새 작업 파일 생성
- 명명 형식: `XXX-description.md` (예: `001-analyze.md`)
- 고수준 명세서, 대상 파일, 수락 기준, 구현 단계 포함
- **변경 작업 시 "## 검증 체크리스트" 섹션 필수 포함 (기존 테스트 통과 + 회귀 검증 시나리오)**
- 예시를 위해 `/tasks` 디렉토리의 마지막 완료된 작업 참조

3. **작업 구현**

- 작업 파일의 명세서를 따름
- 변경 전 반드시 기존 동작을 테스트로 보호
- 각 단계 후 작업 파일 내 진행 상황 업데이트
- 구현 완료 후 기존 테스트 전부 통과 확인
- 테스트 통과 확인 후 다음 단계로 진행
- 각 단계 완료 후 중단하고 추가 지시를 기다림

4. **로드맵 업데이트**

- 로드맵에서 완료된 작업을 ✅로 표시

## 개선 단계

### Phase 1: 현황 분석 및 안전망 구축

- **Task 001: 기존 코드 분석 및 문제점 문서화** - 우선순위
  - 대상 모듈의 현재 구조 및 의존성 파악
  - 코드 냄새, 기술 부채, 성능 병목 목록화
  - 변경 영향 범위 및 위험도 평가

- **Task 002: 기존 동작 보호를 위한 테스트 작성**
  - 현재 public API에 대한 단위 테스트 작성
  - 핵심 비즈니스 로직 회귀 테스트 구성
  - 테스트 커버리지 기준선 확보

### Phase 2: 구조 개선 (내부 리팩토링)

- **Task 003: [핵심 개선 항목 1]** ✅ - 완료
  - See: `/tasks/003-xxx.md`
  - ✅ [세부 구현 사항 1]
  - ✅ [세부 구현 사항 2]
  - ✅ 기존 테스트 전부 통과 확인

- **Task 004: [핵심 개선 항목 2]**
  - [세부 구현 사항 1]
  - [세부 구현 사항 2]
  - 변경 후 회귀 테스트 실행

### Phase 3: 인터페이스 정리 및 호환성 검증

- **Task 005: public API 정리 및 문서화** - 우선순위
  - 불필요한 public 노출 최소화
  - XML 문서 주석 / docstring 정비
  - 하위 호환성 검증 테스트 수행

- **Task 006: 통합 검증**
  - 전체 개선 항목 통합 테스트
  - 기존 파이프라인 / 호출부 호환성 확인
  - 성능 비교 (개선 전/후)

### Phase 4: 마무리 및 기술 부채 정리

- **Task 007: 코드 품질 최종 점검**
  - 정적 분석 도구 실행 및 경고 해소
  - 네이밍 컨벤션 일관성 검토
  - 불필요한 주석 / 데드코드 제거

- **Task 008: 개선 결과 문서화**
  - 변경 사항 요약 문서 작성
  - 개선 전/후 비교 정리
  - 다음 단계 개선 후보 목록 작성
```

### 🎨 작성 지침

### **Phase 구성 원칙 (안전 우선 접근법 기반)**

- **Phase 1: 현황 분석 및 안전망 구축**
    - 기존 코드 구조 및 의존성 정밀 분석
    - 변경 전 기존 동작을 보호하는 테스트 작성
    - 개선 범위 및 위험도 평가
    - 롤백 전략 수립
- **Phase 2: 구조 개선 (내부 리팩토링)**
    - 외부 인터페이스 변경 없이 내부 구조 개선
    - 모듈 분리, 알고리즘 교체, 중복 제거
    - 각 변경 후 즉시 테스트 검증
    - 점진적 개선으로 위험 최소화
- **Phase 3: 인터페이스 정리 및 호환성 검증**
    - public API 최소화 및 명확화
    - 하위 호환성 영향 검증
    - 기존 호출부 통합 테스트
    - 성능 개선 효과 측정
- **Phase 4: 마무리 및 기술 부채 정리**
    - 정적 분석 및 코드 품질 최종 점검
    - 개선 결과 문서화
    - 다음 개선 후보 목록 작성

### **Task 작성 규칙**

1. **명명**: `Task XXX: [동사] + [대상] + [목적]` (예: `Task 001: FeModelGenerator 의존성 분석`)
2. **범위**: 1주 내 완료 가능한 단위로 분해 (개선 작업은 신규 개발보다 작게)
3. **독립성**: 각 Task는 독립적으로 검증 가능하고 롤백 가능하게 구성
4. **구체성**: "개선"이 아닌 "Materials 클래스 분리", "Union-Find 알고리즘 교체" 등 구체적 명시

### **상태 표시 규칙**

- **Phase 상태**:
    - **Phase 제목 + ✅**: 완료된 Phase
    - **Phase 제목만**: 진행 중이거나 대기 중인 Phase
- **Task 상태**:
    - **✅ - 완료**: 완료된 작업 (완료 시 `See: /tasks/XXX-xxx.md` 참조 추가)
    - **우선순위**: 즉시 시작해야 할 작업
    - **상태 없음**: 대기 중인 작업
- **구현 사항 상태**:
    - **✅**: 완료된 세부 구현 사항
    - 일반 리스트: 미완료 세부 구현 사항

### **구현 사항 작성법**

- 각 Task 하위에 3-7개의 구체적 변경 사항 나열
- 대상 클래스/메서드, 변경 방식, 검증 방법 등 실제 개선 요소 포함
- 측정 가능한 완료 기준 제시 (예: "기존 테스트 15개 전부 통과")

### 🚨 품질 체크리스트

생성된 ROADMAP.md가 다음 기준을 만족하는지 확인:

### **📋 기본 요구사항**

- [ ]  PRD의 모든 개선 요구사항이 Task로 분해되었는가?
- [ ]  Task들이 적절한 크기로 분해되었는가? (1주 내 완료 가능)
- [ ]  각 Task의 구현 사항이 구체적이고 실행 가능한가?
- [ ]  전체 로드맵이 실제 개선 작업에서 사용 가능한 수준인가?

### **🏗️ 안전 우선 접근법 준수**

- [ ]  Phase 1에서 현황 분석과 테스트 안전망이 먼저 구성되었는가?
- [ ]  Phase 2에서 내부 구조만 변경하고 외부 인터페이스는 유지되는가?
- [ ]  Phase 3에서 인터페이스 정리 및 호환성 검증이 수행되는가?
- [ ]  각 Task가 독립적으로 롤백 가능한 단위로 구성되었는가?
- [ ]  테스트 작성이 실제 변경보다 먼저 배치되었는가?

### **🔗 의존성 및 순서**

- [ ]  기술적 의존성이 올바르게 고려되었는가?
- [ ]  영향 범위가 큰 변경이 후반 Phase에 배치되었는가?
- [ ]  중복 작업을 최소화하는 순서로 배치되었는가?

### **🧪 검증 체크리스트**

- [ ]  모든 변경 Task에 기존 테스트 통과 기준이 포함되었는가?
- [ ]  각 Task에 "## 검증 체크리스트" 섹션이 명시되었는가?
- [ ]  회귀 테스트 시나리오가 정의되었는가?
- [ ]  하위 호환성 영향 검증이 포함되었는가?
- [ ]  Phase 3에 통합 검증 Task가 포함되었는가?

### 💡 추가 고려사항

- **기존 코드 존중**: 현재 동작하는 코드를 기반으로 점진적 개선
- **호환성**: 기존 호출부와의 인터페이스 계약 유지
- **추적 가능성**: 각 변경이 어떤 요구사항을 해결하는지 명확히 연결
- **문서화**: 개선 이유와 결과를 후임자가 이해할 수 있도록 기록
- **범위 통제**: PRD에 정의된 개선 범위를 벗어나지 않도록 관리
